# 바다낚시 출조 도우미 미니앱 기획서 v2.0

작성일: 2026-07-18 | 배포 플랫폼: 앱인토스 (토스 미니앱, WebView)
v1.0 대비 변경 사유: 플랫폼 정책·API 약관·데이터 소스 실사 결과 반영

## 0. 실현 가능성 판정 (TL;DR)

**결론: 실현 가능하다. 단, v1 기획에서 3가지를 반드시 수정해야 한다.**

| 항목                             | v1 기획            | 실사 결과                                                                                                                              | v2 결정                                      |
| -------------------------------- | ------------------ | -------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| 바람 레이어 1안 (Windy iframe)   | "약관 확인 필요"   | **이중으로 불가.** ① 앱인토스는 iframe 사용 시 보안 심사 반려(유튜브만 예외) ② Windy 약관도 앱/상업 서비스 내 임베드를 명시적으로 금지 | **폐기**                                     |
| 바람 레이어 2안 (Open-Meteo GFS) | "무료"             | Open-Meteo 무료 티어는 **비상업 전용**. 토스 미니앱은 상업 배포로 해석될 여지가 커서 위험                                              | **폐기** (관측·예보 기반 바람 화살표로 대체) |
| 파고 데이터                      | 바다누리 관측 부이 | 관측값만으론 "오늘 오후" 예보 불가. **기상청 단기예보에 파고(WAV) 항목이 포함**되어 있어 예보까지 해결 가능                            | 기상청 단기예보(WAV) + 바다누리 관측 병행    |

추가 발견: 바다누리에 이미 **바다낚시지수 API**(전국 낚시 포인트별 어종·물때·파고·수온 종합 5단계 지수)가 존재한다. 신호등 판단 로직을 밑바닥부터 만들 필요 없이, 이 지수를 핵심 입력으로 쓰고 기상특보·풍속으로 보정하는 방식이 개발량도 적고 방어 논리(공공기관 지수 기반)도 좋다.

## 1. 개요 (v1 유지, 요약)

물때(조석)와 현장 기상(파고·바람·날씨)을 한 화면에서 확인하고, "오늘 출조해도 되는가?"라는 **판단**까지 제공하는 토스 미니앱. 차별점은 ① 무설치(토스 3,000만 유저), ② 출조 신호등이라는 해석 계층, ③ 의도적으로 가벼운 사용성.

한 줄 정의: **"토스 열고 10초 안에 오늘 출조 여부를 판단하게 해주는 앱"** (유지)

## 2. 확정 데이터 소스 (v2)

모든 API는 공공데이터포털(data.go.kr) 활용신청 → 인증키 발급 방식. 무료.

| 데이터                                          | API                              | 엔드포인트 (확인됨)                     | 용도                                                                      |
| ----------------------------------------------- | -------------------------------- | --------------------------------------- | ------------------------------------------------------------------------- |
| 만조/간조 시각·조위                             | 국립해양조사원 조석예보(고,저조) | `apis.data.go.kr/1192136/tideFcst...`   | 홈 카드, 물때표                                                           |
| 24시간 조위 곡선                                | 국립해양조사원 조석예보(시계열)  | `apis.data.go.kr/1192136/tideFcstTime`  | 포인트 상세 그래프                                                        |
| 낚시 종합 지수                                  | 국립해양조사원 **바다낚시지수**  | `apis.data.go.kr/1192136/fcstFishingv2` | **신호등 핵심 입력** (갯바위/선상 구분, 어종별 지수, 물때·파고·수온 포함) |
| 파고·수온 관측(실황)                            | 바다누리 조위관측소·해양관측부이 | 바다누리 오픈API                        | 홈 "지금 현장" 카드                                                       |
| 날씨·기온·강수확률·풍향/풍속·**파고(WAV)** 예보 | 기상청 단기예보                  | 공공데이터포털                          | 시간대별 예보, 신호등 보정                                                |
| 기상특보 (풍랑주의보 등)                        | 기상청 기상특보 통보문           | 공공데이터포털                          | 경고 배너, 무조건 '비추천' 처리                                           |

**삭제된 소스**: Windy(임베드 금지), Open-Meteo(비상업 한정).

주의사항:

- 공공데이터포털 API는 CORS 헤더가 없고 키 노출 문제가 있으므로 **프록시 백엔드 필수** (v1 판단 유효).
- 활용신청 시 "개발계정 → 운영계정 승인" 단계가 있으므로 트래픽 증가 전에 운영계정 전환 신청 필요.
- 물때(몇 물) 계산: 바다낚시지수 응답에 물때 정보가 포함되며, 없으면 음력 날짜 기반 계산식(음력일 ±로 1물~조금 매핑)으로 백엔드에서 산출 — 외부 의존 없음.

## 3. 출조 신호등 로직 v2

v1의 "파고/풍속/강수확률 임계값" 자체 로직은 유지하되, 판단의 뼈대를 바꾼다.

```
1) 기상특보(풍랑·강풍·태풍 등) 발효 중  → 무조건 빨강 + 최상단 경고 배너
2) 바다낚시지수(5단계)를 기본값으로 매핑
   - 매우좋음·좋음 → 초록 / 보통 → 노랑 / 나쁨·매우나쁨 → 빨강
3) 기상청 단기예보로 보정 (지수가 커버 못 하는 시간대 변화 반영)
   - 예보상 풍속 9m/s 초과 또는 파고 1.0m 초과 시간대 존재 → 한 단계 강등
   - 강수확률 60% 초과 → 한 단계 강등
4) 근거 문구 생성: "출조하기 좋아요 · 7물 · 파고 낮음" / "주의 · 오후부터 풍속 8m/s"
```

장점: ① 공공기관 지수를 근거로 삼아 안전 책임 논란 방어에 유리 ② 어종별 지수가 응답에 포함되어 백로그(어종별 시즌 안내)의 기반이 됨 ③ 자체 임계값은 "보정" 역할로 축소되어 튜닝 부담 감소.

안전 고지 "본 정보는 참고용이며 실제 출조 시 기상특보를 반드시 확인하세요" 상시 표기 (v1 유지).

## 4. 지도 결정 (v2)

### 4.1 베이스맵: Leaflet + VWorld 타일

| 후보                               | 판단                                                                              |
| ---------------------------------- | --------------------------------------------------------------------------------- |
| **VWorld(국토부 브이월드) — 채택** | 무료(키 발급제), 한글 지명, 국내 서비스 실적 다수, XYZ 타일로 Leaflet에 바로 연결 |
| OpenStreetMap 공식 타일            | tile.openstreetmap.org는 프로덕션 대량 사용을 정책상 금지. 폴백으로만             |
| 네이버/카카오 지도 SDK             | 무료 쿼터는 있으나 SDK 무게·약관 관리 부담. 미채택                                |

### 4.2 바람 표현: 애니메이션 레이어 → 화살표 마커

Windy식 파티클 애니메이션 레이어는 데이터 소스(합법·무료)가 없으므로 MVP에서 제외한다. 대신:

- 지도 위 관측소·포인트 핀에 **풍향 화살표 + 풍속 숫자**를 마커로 표시 (기상청 예보 + 부이 관측 기반)
- 파고는 핀 색상(초록/노랑/빨강)으로 인코딩 → 지도 자체가 "신호등 지도"가 됨

이 방식이 오히려 컨셉("10초 판단")에 더 부합한다. Windy 수준의 시각화는 우리의 승부처가 아니다(승부처는 해석 계층). 파티클 레이어가 정말 필요해지면 2차에서 기상청 격자 예보 데이터로 leaflet-velocity 자체 구현을 검토한다(데이터가 공공이므로 합법).

### 4.3 핀치줌

앱인토스는 핀치줌을 원칙적으로 금지하나 **지도는 예외 인정**. 지도 컨테이너에만 제스처 허용, 나머지 화면은 `user-scalable=no` 유지.

## 5. 화면 구성 및 UI/UX (v2)

### 5.1 전제: TDS 필수

비게임 미니앱은 TDS(토스 디자인 시스템) 적용이 검수 요건이다. 현재 프로젝트에 `@toss/tds-mobile`이 이미 설치되어 있어 그대로 사용한다. TDS의 카드·리스트·바텀시트·배지 컴포넌트로 아래 화면을 조립하면 "토스 안의 네이티브 같은" 깔끔한 UI가 가능하다. **깔끔한 UI/UX는 충분히 실현 가능하며, 오히려 TDS를 쓰는 것이 커스텀 디자인보다 빠르고 검수도 안전하다.**

### 5.2 탭 구조: 4탭 → 3탭 (YAGNI 적용)

v1의 "즐겨찾기 탭"은 삭제한다. 즐겨찾기(최대 5개)는 홈 상단의 포인트 전환 칩(가로 스와이프)으로 충분하며, 관리(추가/삭제)는 지도와 상세 화면에서 이루어진다. 탭이 하나 줄면 개발량과 인지 부하가 모두 준다.

```
[홈]              [지도]             [물때표]
오늘 요약          포인트 탐색         출조일 계획
─────────────────────────────────────────────
· 포인트 전환 칩    · VWorld 베이스맵   · 주간/월간 리스트
· 출조 신호등 카드  · 신호등 색 핀      · 날짜별 몇물·만간조
  (근거 한 줄)     · 풍향 화살표       · 월령 아이콘
· 만조/간조 시각   · 탭→바텀시트 요약
· 몇 물 · 월령       + 즐겨찾기 버튼
· 파고/풍향풍속
· 날씨/수온
· (특보 시) 경고배너
```

포인트 상세(홈 카드 탭 or 바텀시트 "자세히"): 24시간 조위 곡선(현재 시각 마커), 시간대별 파고·바람·날씨 예보 타임라인, 즐겨찾기 토글, 안전 고지 문구.

### 5.3 첫 실행 흐름 (v1 유지 + SDK 확정)

첫 실행 → 위치 권한(`@apps-in-toss/web-framework`의 geolocation, `granite.config.ts` permissions에 등록) → 최근접 조석 예보지점을 임시 기본값으로 홈 즉시 렌더 → 지도/검색으로 탐색 → 즐겨찾기 등록. 권한 거부 시 기본 포인트(예: 인천)로 폴백하고 검색 유도.

- 즐겨찾기 저장: `localStorage` (WebView에서 동작). 기기 간 동기화가 필요해지면 2차에서 `getAnonymousKey`(비게임 전용 사용자 식별키) + 백엔드 저장 검토.
- 24시간 조위 곡선: 경량 SVG 차트 직접 구현 또는 초경량 차트 라이브러리. 번들 100MB 제한과는 무관하나 로딩 속도를 위해 무거운 차트 라이브러리는 지양.

## 6. 아키텍처 (v2)

```
토스 앱 (WebView)
  └─ 미니앱 프론트: Vite + React 18 + TDS(@toss/tds-mobile)
       + Leaflet(VWorld 타일)  ← 이미 스캐폴드 완료된 현 저장소
        │  HTTPS (CORS 허용은 백엔드에서)
        ▼
  경량 백엔드 (프록시 + 캐시 + 신호등 계산)
        │
        ├─ 바다누리/공공데이터포털 (조석·낚시지수·부이)
        └─ 기상청 (단기예보·기상특보)
```

백엔드 권장: **Cloudflare Workers** (Node.js 서버 대비 운영 부담·비용이 0에 가깝고, Cache API/KV로 캐싱이 쉬움).

- 조석예보: 날짜·지점별로 사실상 불변 → 24시간 캐시
- 낚시지수·단기예보: 1~3시간 캐시
- 기상특보: 10분 캐시
- 신호등 판단은 백엔드에서 계산해 프론트에는 `{level, reason}` 으로 내려줌 → 로직 수정 시 앱 재배포(재검수) 불필요

관측소 매핑: 조석 예보지점·부이·기상청 격자(nx,ny)가 서로 다른 좌표 체계이므로, 포인트(관측소) 마스터 테이블을 정적 JSON으로 1회 구축해 백엔드에 내장한다. MVP는 전국 조석 예보지점(약 180개) 기준.

## 11. 열린 결정 사항

1. **갯바위 vs 선상 구분 UI**: 바다낚시지수 API가 갯바위/선상을 구분해 제공. MVP에서 토글로 노출할지, 갯바위 고정으로 시작할지. → 권장: 갯바위 기본 + 설정에서 전환.
2. **백엔드 플랫폼**: Cloudflare Workers 권장이나, 기존 Node.js 인프라 선호 시 Fastify+메모리 캐시도 무방.
3. **차트 구현**: 조위 곡선을 SVG 자체 구현(가벼움) vs 라이브러리(빠른 개발). → 권장: 자체 SVG (단일 곡선이라 난이도 낮음).
